# Product Manager Brief Writing

Expert guidance on writing clear, concise, and actionable briefs for product development.

## When to Use This Skill

- Writing Product Requirements Documents (PRDs)
- Creating feature briefs and specifications
- Drafting user stories and acceptance criteria
- Creating design briefs for designers
- Writing technical briefs for engineering
- Drafting executive summaries and strategy documents

## Core Principles

### 1. Clarity Over Completeness
- Start with the "why" and "what" before the "how"
- Use simple, direct language
- Define technical terms and acronyms

### 2. Action-Oriented
- Include clear success metrics
- Define specific deliverables
- Assign ownership explicitly

### 3. User-Centric
- Start with user problems, not solutions
- Include user research and data
- Use personas and scenarios

### 4. Data-Driven
- Support decisions with evidence
- Include relevant metrics and KPIs
- Quantify impact and success criteria

## Quick Reference

### Choose Template by Purpose
| Template | Use For |
|----------|---------|
| **PRD** | Comprehensive feature development |
| **Feature Brief** | Quick lightweight specs |
| **User Story** | Development work items |
| **Design Brief** | Design kickoff |
| **Technical Brief** | Engineering requirements |
| **Executive Brief** | Leadership decisions |

### The BLUF Principle
**Always start with:**
1. What is this?
2. Why does it matter?
3. What do you need?

### Writing for Audiences

| Audience | Focus On |
|----------|----------|
| **Executives** | Business impact, ROI, recommendations |
| **Engineering** | Technical specs, edge cases, constraints |
| **Design** | User problems, context, desired outcomes |

### Brief Checklist
**Content:**
- [ ] Clear problem statement
- [ ] Evidence/data
- [ ] Specific goals and metrics
- [ ] Defined scope (in/out)
- [ ] Identified risks
- [ ] Clear next steps

**Clarity:**
- [ ] Audience-appropriate language
- [ ] Jargon explained
- [ ] Concise sentences
- [ ] Active voice
