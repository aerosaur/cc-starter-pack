## The BLUF Principle

**Bottom Line Up Front** - Always lead with the key message.

### BLUF Structure
1. **What** - The recommendation/request
2. **Why** - The business impact
3. **Ask** - What you need from the reader

---

## Writing for Different Audiences

### For Executives
- Lead with business impact
- High-level metrics only
- Clear recommendations
- Concise (1-2 pages max)
- ROI and payback period
- Risks with mitigations

### For Engineering
- Specific requirements
- Technical constraints
- Edge cases listed
- Detailed specs
- Performance requirements
- API contracts

### For Design
- User problems and context
- Desired outcomes
- Research and data
- Constraints clearly stated
- Success metrics
- User personas

## Comprehensive Brief Checklist

### Content
- [ ] Clear problem statement with user impact
- [ ] Evidence/data supporting the problem
- [ ] Specific, measurable goals
- [ ] Defined scope (in/out)
- [ ] Identified risks with mitigations
- [ ] Clear next steps and owners
- [ ] Acronyms defined or linked

### Structure
- [ ] Executive summary at top
- [ ] Logical flow (problem â†’ solution â†’ plan)
- [ ] Scannable headers
- [ ] Visual hierarchy (bold, bullets, tables)
- [ ] Supporting links (not inline detail)

### Clarity
- [ ] Audience-appropriate language
- [ ] Jargon explained
- [ ] Concise sentences (no fluff)
- [ ] Active voice
- [ ] No ambiguous statements

### Actionability
- [ ] Clear recommendation
- [ ] Decision options listed
- [ ] Timeline specified
- [ ] Owners assigned
- [ ] Success criteria defined

---

## Key Principles Summary

Great briefs are:
- **Clear:** Easy to understand
- **Concise:** Respect reader's time
- **Complete:** All necessary info included
- **Actionable:** Enable decisions
- **User-centric:** Start with user value
- **Data-driven:** Evidence-based

**Remember:** You're writing to enable action, not demonstrate knowledge.
